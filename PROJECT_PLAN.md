# YouTube 长视频翻译器一期规划

## 1. 一期目标

做一个面向 30-120 分钟英文演讲/访谈视频的本地优先翻译工具，核心能力是：

- 输入 YouTube 链接
- 自动获取音频
- 自动转写英文字幕
- 自动翻译成中文
- 输出双语字幕、中文字幕、全文稿
- 在网页中预览字幕和时间轴

一期不把“中文配音”作为主目标，只预留升级接口。

## 2. 为什么不是浏览器插件优先

建议一期交付为“本地 Web App”，而不是 Chrome 插件。

原因：

- 长视频处理是重任务，适合后台队列，不适合插件前台脚本硬扛
- 需要 `yt-dlp`、`ffmpeg`、音频切分、缓存、导出字幕，插件形态不顺手
- API Key 放在插件端风险更高
- 后续接入本地模型和批量任务时，桌面/本地服务形态扩展性更好

建议的交互形态：

- 用户打开本地页面，例如 `http://localhost:3000`
- 粘贴 YouTube URL
- 选择目标语言、输出格式、翻译策略
- 提交任务并查看进度
- 完成后下载 `srt` / `ass` / `txt` / `json`

如果后面需要“一键从 YouTube 页面发送到翻译器”，二期再加轻量插件，只负责把当前 URL 发给本地服务。

## 3. 一期推荐架构

### 前端

- `Next.js` 或 `React + Vite`
- 作用：任务创建、进度展示、字幕预览、导出下载

### 后端

- `Python + FastAPI`
- 作用：下载音频、转写、分段、翻译、导出、任务调度

### 任务层

- 简单起步可用 FastAPI 后台任务
- 稳一点可用 `RQ` / `Celery` / `Arq`
- 一期先不引入太重的消息队列，先做单机任务队列即可

### 关键工具

- `yt-dlp`：拉取 YouTube 音频和元数据
- `ffmpeg`：抽音频、重采样、切片、混流
- 转写引擎：优先 API 模式，保留本地模式接口
- 翻译引擎：优先 `DeepSeek`

## 4. 模型策略

### 4.1 推荐的职责拆分

- 转写：不要指望 DeepSeek，单独选转写引擎
- 翻译：主力使用 `DeepSeek`
- 术语统一 / 人名纠错 / 总结：也使用 `DeepSeek`
- 配音：暂不做主线，只预留 TTS 接口

### 4.2 为什么 DeepSeek 不适合吃完整链路

我当前查到的 DeepSeek 官方 API 文档重点是 OpenAI 兼容的聊天接口和定价页；没有看到可直接用于长音频转写或 TTS 的官方音频 API 文档。因此，一期更现实的做法是：

- `ASR` 用专业转写方案
- `Translation + Post-edit` 用 DeepSeek

### 4.3 DeepSeek 适合放在哪一步

DeepSeek 很适合做下面这些文本任务：

- 把英文字幕段翻成中文
- 保留时间轴不变，只替换文本
- 做术语表统一
- 修正常见的 ASR 错词
- 生成双语对照稿
- 长文总结与章节提要

## 5. 成本策略

### 5.1 一期默认策略

采用“低成本混合方案”：

- 转写：先用稳定 ASR 服务，或者给用户可选本地转写
- 翻译：使用 DeepSeek
- 存储：本地文件缓存，避免云存储成本

### 5.2 当前查到的 DeepSeek 官方价格

截至 2026-03-18，我查到 `deepseek-chat` / `deepseek-reasoner` 的官方定价页面显示：

- 输入 tokens（cache miss）：`$0.28 / 1M`
- 输入 tokens（cache hit）：`$0.028 / 1M`
- 输出 tokens：`$0.42 / 1M`

对长视频翻译来说，真正的大头通常不是“翻译”，而是“转写”。

### 5.3 粗略成本判断

对 1 小时英语访谈：

- 翻译成本通常会低到可以忽略
- 转写成本才是主成本

所以如果我们要追求性价比，最重要的优化顺序是：

1. 先把转写成本打下来
2. 再把翻译做稳定
3. 最后才考虑配音

## 6. 转写方案建议

### 方案 A：云端转写优先

优点：

- 开发快
- 质量稳定
- 省去本地模型部署复杂度

缺点：

- 有持续 API 成本
- 长视频累计成本会上升

适合：

- 先快速做出 MVP

### 方案 B：本地转写优先

候选：

- `faster-whisper`
- `whisper.cpp`

优点：

- 边际成本低
- 更适合重度使用

缺点：

- 对用户机器性能更敏感
- Windows 环境安装链更复杂
- 无 GPU 时速度可能不理想

适合：

- 你自己先用
- 追求长期低成本

### 一期建议

先实现“双引擎架构”：

- 默认支持 API 转写
- 预留本地 `Whisper` 转写接口

这样我们先把产品跑通，再慢慢把成本压低。

## 7. 一期功能清单

### 必做

- 输入 YouTube 链接
- 拉取视频标题、时长、封面
- 下载音频
- 生成英文转写
- 生成中文字幕
- 生成中英双语字幕
- 导出 `srt`
- 导出全文 `txt`
- 任务进度和失败重试
- 本地缓存，避免重复处理同一个视频

### 应该做

- 支持术语表
- 支持翻译风格选择
- 支持分段重翻
- 支持字幕预览和时间轴编辑
- 支持导出 `json`

### 二期再做

- 中文配音
- 保留原声并自动混音
- 多说话人识别
- 章节摘要
- 插件一键发送 YouTube 当前页面

## 8. 一期技术模块拆分

### 模块 1：任务输入

- 接受 YouTube URL
- 校验 URL
- 抓取视频基础信息

### 模块 2：媒体处理

- 调用 `yt-dlp` 下载音频
- 调用 `ffmpeg` 转为统一采样率
- 超长音频自动切段

### 模块 3：转写

- 统一转写接口
- 支持多个 provider
- 输出标准化段落结构：
  - `start`
  - `end`
  - `text`
  - `speaker?`
  - `confidence?`

### 模块 4：翻译

- 按字幕段批量翻译
- 保护时间轴
- 限制单批长度
- 失败自动重试
- 用术语表做后处理

### 模块 5：导出

- 导出英文 `srt`
- 导出中文 `srt`
- 导出双语 `srt`
- 导出 `txt`
- 导出标准 `json`

### 模块 6：缓存与历史

- 本地数据库记录任务
- 缓存下载后的音频和中间产物
- 避免重复计费

## 9. 数据结构建议

一期统一输出一个内部 `TranscriptDocument`：

```json
{
  "video_id": "xxx",
  "source_lang": "en",
  "target_lang": "zh-CN",
  "segments": [
    {
      "id": 1,
      "start": 0.0,
      "end": 3.2,
      "source_text": "Hello everyone.",
      "translated_text": "大家好。",
      "speaker": null,
      "confidence": null
    }
  ]
}
```

这个结构可以让我们后面平滑升级到：

- 多语言
- 多说话人
- 配音
- 人工校对

## 10. 推荐的开发顺序

### Sprint 1

- 初始化前后端项目
- 接通 YouTube URL 输入
- 接通 `yt-dlp`
- 接通单一转写 provider
- 接通 DeepSeek 翻译
- 导出中文 / 双语 `srt`

### Sprint 2

- 增加任务历史
- 增加缓存
- 增加失败重试
- 优化长视频分段策略
- 增加字幕预览

### Sprint 3

- 增加本地 Whisper 模式
- 增加术语表
- 增加导出 `json` / `txt`
- 增加简单编辑能力

## 11. 当前环境观察

在当前机器上我已经确认：

- `Node.js` 可用
- `Python 3.12` 可用
- `yt-dlp` 可用
- `ffmpeg` 当前未安装

这意味着如果现在开工，一件最先要补的基础依赖就是 `ffmpeg`。

## 12. 我建议你现在确认的几个问题

### 必须确认

1. 一期是否只做 `英文 -> 中文（简体）`？
2. 一期是否只支持 YouTube URL，不处理本地视频文件？
3. 一期是否接受“先做字幕翻译，不做中文配音”？
4. 一期是否接受“本地运行”，而不是直接做成在线网站？

### 建议确认

1. 你更看重“开发快上线”，还是“后续单次使用成本更低”？
2. 你自己机器是否有可用 GPU？
3. 你是否已经有某个转写 API 的 key？

## 13. 最终建议

最稳的一期路线是：

`本地 Web App + Python 后端 + yt-dlp + ffmpeg + 专业 ASR + DeepSeek 翻译`

产品定位不是“网页即时翻译插件”，而是“长视频字幕翻译工作台”。

这样成功率最高、成本可控、后续也最容易升级到配音。
