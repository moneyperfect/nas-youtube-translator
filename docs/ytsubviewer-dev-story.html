起因

一个很私人的需求

我每周都要看很多英文 YouTube 视频，主要是技术讲座和产品发布。字幕翻译是个刚需，但现有工具让我很不爽。

DeepL 翻译出来的字幕像是机翻味的散文，读起来很累。专门的视频翻译 SaaS 又太贵，而且数据要上传到别人服务器。

最让我受不了的是，每次翻译完都要手动调格式。SRT 文件的时间轴经常对不上，双语字幕挤在一起根本看不清。

过程

三个周末的折腾

第一个周末

把链路跑通

用 yt-dlp 下载视频，faster-whisper 转写，DeepSeek 翻译，ffmpeg 烧字幕。很粗糙，但能跑。最头疼的是字幕时间轴对齐，调了整整一下午。

第二个周末

从脚本变成工具

加了 Web 界面，FastAPI 后端，任务队列。最意外的是"字幕编辑器"——本来没想做，但翻译完发现有几句话翻得不对，就顺手加了单句重译功能。

第三个周末

强迫症式打磨

代码重构了三遍，UI 重写了两遍。加了配置持久化、批量处理、质量报告。最后还写了 65 个单元测试——对一个"自己用的工具"来说有点过分了。

技术

堆了哪些东西

技术栈不复杂，但每个环节都踩了坑。比如 faster-whisper 在 Windows 上的 CUDA DLL 路径问题，我查了两天才搞定。

Python FastAPI yt-dlp faster-whisper ffmpeg DeepSeek API 原生 JS

收获

几个意外的发现

1. 工具做好了，自己用起来真的很爽

以前翻译一个 10 分钟的视频要半小时，现在粘贴链接等两分钟就完事了。这种"为自己的痛点做东西"的满足感，比给老板写代码强太多了。

2. AI 能写代码，但不能替你思考

整个项目 90% 的代码是 AI 写的，但架构设计、功能取舍、体验细节都是我自己想的。AI 是执行者，不是决策者。

3. "够用"和"好用"之间差了一个周末

第一个周末就能用了，但很粗糙。后面两个周末花在打磨体验上——加载动画、Toast 提示、配置持久化。这些"不重要"的东西，恰恰决定了你会不会继续用它。

最好的工具，是那种你做完之后
再也回不去手动操作的工具。
